PHẦN 1: ĐỊNH NGHĨA CALL API

1.1. API là gì? - Giải thích từ cơ bản đến nâng cao

Khái niệm cơ bản

API (Application Programming Interface) là một tập hợp các định nghĩa và giao thức cho phép các ứng dụng phần mềm giao tiếp với nhau. Nó hoạt động như một "người trung gian" giúp các ứng dụng trao đổi dữ liệu và chức năng một cách an toàn và có kiểm soát.

Ví dụ thực tế (Analogies)

Ví dụ 1: Nhà hàng (RESTAURANT)
  • Thực khách (Client): Bạn (ứng dụng frontend)
  • Nhà bếp (Server): Nơi chế biến món ăn (backend, database)
  • Người phục vụ (API): Người nhận order và mang món ra
  • Thực đơn (API Documentation): Danh sách các món có thể gọi

Quy trình:

  • Bạn xem thực đơn (đọc tài liệu API)
  • Bạn gọi người phục vụ (gửi request)
  • Người phục vụ ghi order và chuyển xuống bếp (xử lý request)
  • Đầu bếp nấu món ăn (server xử lý logic)
  • Người phục vụ mang món ra cho bạn (gửi response)
Ví dụ 2: Ngân hàng (BANK)
  • Bạn không thể tự ý mở két sắt của ngân hàng để lấy tiền
  • Bạn phải đến quầy giao dịch (API), điền phiếu yêu cầu (request), xuất trình CMND (authentication)
  • Giao dịch viên kiểm tra và đưa tiền cho bạn (response)

1.2. Call API là gì? - Định nghĩa chính xác

Call API (Gọi API) là hành động mà một ứng dụng (client) thực hiện để gửi một yêu cầu (HTTP request) đến một máy chủ (server) thông qua một URL cụ thể, nhằm yêu cầu server thực hiện một tác vụ và trả về kết quả.

Ví dụ code minh họa (JavaScript - Fetch API):

JavaScript - Fetch API
// ĐÂY LÀ MỘT CALL API
fetch('https://jsonplaceholder.typicode.com/users/1')
  .then(response => response.json())
  .then(data => {
    console.log('Dữ liệu nhận được:', data);
    // { id: 1, name: 'Leanne Graham', email: 'Sincere@april.biz', ... }
  })
  .catch(error => {
    console.error('Lỗi khi gọi API:', error);
  });

1.3. Các thành phần của một Call API

Một cuộc gọi API đầy đủ bao gồm 5 thành phần chính:

1.3.1. Endpoint (Điểm cuối)

Là URL cụ thể mà client gửi request đến.

Cấu trúc: protocol://domain:port/path?query=value

Ví dụ: https://api.example.com:443/v1/users?page=2

  • https:// : Giao thức
  • api.example.com : Domain
  • :443 : Port (thường 80 cho HTTP, 443 cho HTTPS)
  • /v1/users : Path (đường dẫn)
  • ?page=2 : Query parameters

1.3.2. Method (Phương thức HTTP)

Hành động muốn thực hiện trên server:

Method Mục đích CRUD tương ứng
GET Lấy dữ liệu READ
POST Tạo mới dữ liệu CREATE
PUT Cập nhật toàn bộ UPDATE (thay thế)
PATCH Cập nhật một phần UPDATE (chỉnh sửa)
DELETE Xóa dữ liệu DELETE

1.3.3. Headers (Tiêu đề)

Headers chứa metadata về request/response. Định dạng key-value.

Headers thường gặp:

Headers
{
  "Content-Type": "application/json", // Định dạng dữ liệu gửi lên
  "Accept": "application/json", // Định dạng dữ liệu muốn nhận về
  "Authorization": "Bearer eyJhbGciOiJIUzI1...", // Token xác thực
  "User-Agent": "MyApp/1.0", // Thông tin client
  "Cache-Control": "no-cache" // Kiểm soát cache
}

1.3.4. Body (Nội dung - Request Body)

Dữ liệu gửi lên server (thường dùng với POST, PUT, PATCH).

Định dạng phổ biến: JSON, form-data, x-www-form-urlencoded

Ví dụ JSON:

JSON
{
  "name": "Nguyễn Văn A",
  "email": "a.nguyen@example.com",
  "age": 25,
  "address": {
    "city": "Hà Nội",
    "district": "Cầu Giấy"
  }
}

1.3.5. Authentication (Xác thực)

Các phương thức xác thực phổ biến:

  • API Key: X-API-Key: abc123def456
  • Basic Auth: Authorization: Basic base64(username:password)
  • Bearer Token (JWT): Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
  • OAuth 2.0: Phức tạp hơn, dùng cho các ứng dụng lớn (Google, Facebook login)

1.4. Phân loại API

Theo kiến trúc

  • REST API (Representational State Transfer)
    • Phổ biến nhất hiện nay
    • Sử dụng HTTP methods chuẩn
    • Stateless (không lưu trạng thái)
    • Trả về JSON/XML
  • SOAP API (Simple Object Access Protocol)
    • Giao thức chặt chẽ, dùng XML
    • Bảo mật cao hơn (WS-Security)
    • Thường dùng trong ngân hàng, thanh toán
  • GraphQL API
    • Do Facebook phát triển
    • Client tự chọn fields muốn lấy
    • Giảm over-fetching/under-fetching
  • WebSocket API
    • Kết nối hai chiều, realtime
    • Dùng trong chat, game, stock market

Theo phạm vi sử dụng

  • Public API (Open API): Mở cho tất cả mọi người (Google Maps, Facebook)
  • Private API (Internal API): Chỉ dùng nội bộ công ty
  • Partner API: Chia sẻ với đối tác cụ thể
  • Composite API: Kết hợp nhiều API trong một lần gọi

1.5. Thuật ngữ quan trọng

1.5.1. REST (Representational State Transfer)

Là một kiến trúc, không phải giao thức. RESTful API tuân theo 6 nguyên tắc:

  1. Client-Server: Tách biệt giao diện và dữ liệu
  2. Stateless: Mỗi request độc lập, không phụ thuộc request trước
  3. Cacheable: Response có thể được cache
  4. Uniform Interface: Giao diện đồng nhất
  5. Layered System: Hệ thống phân tầng
  6. Code on Demand (tùy chọn): Server có thể gửi code về client

1.5.2. JSON (JavaScript Object Notation)

Định dạng dữ liệu nhẹ, dễ đọc, dễ parse.

JSON
{
  "users": [
    {
      "id": 1,
      "name": "John Doe",
      "email": "john@example.com",
      "isActive": true
    },
    {
      "id": 2,
      "name": "Jane Smith",
      "email": "jane@example.com",
      "isActive": false
    }
  ],
  "total": 2,
  "page": 1
}

1.5.3. XML (eXtensible Markup Language)

Định dạng cũ hơn, verbose hơn JSON.

XML
<users>
  <user id="1">
    <name>John Doe</name>
    <email>john@example.com</email>
    <isActive>true</isActive>
  </user>
  <user id="2">
    <name>Jane Smith</name>
    <email>jane@example.com</email>
    <isActive>false</isActive>
  </user>
  <total>2</total>
  <page>1</page>
</users>

1.5.4. SDK (Software Development Kit)

Bộ công cụ phát triển phần mềm, bao gồm:

  • Thư viện code (library)
  • Tài liệu (documentation)
  • Code mẫu (sample code)
  • Công cụ debug

Ví dụ: Firebase SDK, Facebook SDK, Stripe SDK

1.5.5. Các thuật ngữ khác

Thuật ngữ Ý nghĩa
Endpoint Địa chỉ cụ thể của API
Payload Dữ liệu trong request/response
Rate Limit Giới hạn số lần gọi API trong khoảng thời gian
Pagination Phân trang dữ liệu (page, limit, offset)
Idempotent Gọi nhiều lần cho kết quả giống nhau (GET, PUT, DELETE)
CORS Cơ chế bảo mật của trình duyệt
Webhook API ngược - server gọi về client khi có sự kiện

PHẦN 2: CÁCH THỨC HOẠT ĐỘNG CỦA CALL API

2.1. Mô hình Client-Server

Định nghĩa

Mô hình Client-Server là kiến trúc mạng phân tán, trong đó:

  • Client: Thiết bị/ứng dụng yêu cầu dịch vụ (gửi request)
  • Server: Máy tính cung cấp dịch vụ (xử lý và trả response)

Đặc điểm chi tiết

Client có thể là:

  • Web browser (Chrome, Firefox)
  • Mobile app (iOS, Android)
  • Desktop app (Electron, .NET)
  • IoT device (smart watch, smart TV)
  • Another server (trong microservices)

Server bao gồm:

  • Web server (Nginx, Apache) - nhận request
  • Application server (Node.js, Python Django, Java Spring) - xử lý logic
  • Database server (MySQL, MongoDB) - lưu trữ dữ liệu

Sơ đồ luồng hoạt động

2.2. Quy trình 5 bước của một Call API

Bước 1: Xác định nhu cầu

Client cần thực hiện một tác vụ:

  • Hiển thị danh sách sản phẩm
  • Đăng nhập user
  • Cập nhật thông tin profile
  • Thanh toán đơn hàng

Bước 2: Cấu hình Request

Lập trình viên (hoặc framework) xây dựng HTTP request

Bước 3: Gửi Request

Quá trình diễn ra trong mạng:

  • DNS Resolution: Chuyển domain api.example.com thành IP (VD: 104.18.22.111)
  • TCP Handshake: Thiết lập kết nối 3 bước (SYN, SYN-ACK, ACK)
  • TLS Handshake (nếu HTTPS): Thiết lập mã hóa SSL/TLS
  • Gửi dữ liệu: HTTP request được chia thành các gói tin TCP/IP
  • Routing: Các gói tin đi qua nhiều router, switch đến server

Bước 4: Xử lý Request tại Server

Server nhận request và xử lý theo quy trình

Bước 5: Nhận và xử lý Response

Client nhận response và xử lý:

Xử lý Response
fetch("https://jsonplaceholder.typicode.com/users")
  .then(response => response.json())
  .then(data => {
    console.log(data);
  })
  .catch(error => {
    console.log("Error:", error);
  });

2.3. HTTP Protocol - Giao thức vận chuyển

HTTP là gì?

HTTP (Hypertext Transfer Protocol) là giao thức truyền tải siêu văn bản, hoạt động dựa trên mô hình request-response giữa client và server.

Đặc điểm

  • Port mặc định: 80 (HTTP), 443 (HTTPS)
  • Stateless: Mỗi request độc lập
  • Text-based: Con người có thể đọc được (khác với binary protocol)
  • Extensible: Có thể thêm headers tùy chỉnh

2.4. Status Codes - Mã trạng thái HTTP chi tiết

2xx: Thành công

Code Tên Ý nghĩa Ví dụ
200 OK Thành công, có response body GET thành công
201 Created Tạo mới thành công POST tạo user
202 Accepted Đã chấp nhận nhưng chưa xử lý xong Xử lý bất đồng bộ
204 No Content Thành công, không có body DELETE thành công

3xx: Chuyển hướng

Code Tên Ý nghĩa Ví dụ
301 Moved Permanently Chuyển hướng vĩnh viễn Thay đổi domain
302 Found Chuyển hướng tạm thời Redirect sau login
304 Not Modified Dùng cache Kiểm tra ETag

4xx: Lỗi từ Client

Code Tên Ý nghĩa Nguyên nhân
400 Bad Request Request không hợp lệ Thiếu field, sai format
401 Unauthorized Chưa xác thực Thiếu token, token hết hạn
403 Forbidden Không có quyền Đã login nhưng không đủ quyền
404 Not Found Không tìm thấy resource URL sai, ID không tồn tại
405 Method Not Allowed Sai phương thức Dùng POST thay vì GET
409 Conflict Xung đột dữ liệu Email đã tồn tại
422 Unprocessable Entity Dữ liệu không hợp lệ Validation lỗi
429 Too Many Requests Vượt quá rate limit Gọi API quá nhiều

5xx: Lỗi từ Server

Code Tên Ý nghĩa Nguyên nhân
500 Internal Server Error Lỗi server Exception không xử lý
502 Bad Gateway Lỗi gateway Proxy nhận response lỗi
503 Service Unavailable Quá tải/bảo trì Server đang quá tải
504 Gateway Timeout Timeout Request quá lâu

PHẦN 3: CÁC PHƯƠNG THỨC CALL API PHỔ BIẾN

3.1. GET - Lấy dữ liệu

Đặc điểm

  • Mục đích: Lấy dữ liệu từ server
  • Idempotent: Gọi 1 lần hay 100 lần đều cho kết quả giống nhau
  • Cacheable: Có thể cache được
  • Không có body (hoặc không nên có)
  • Dữ liệu gửi qua: URL parameters, query string

Cú pháp

GET Request
// Lấy tất cả users
fetch('https://api.example.com/users')

// Lấy user cụ thể theo ID
fetch('https://api.example.com/users/123')

// Lọc, phân trang, sắp xếp
fetch('https://api.example.com/users?page=2&limit=10&sort=name&order=asc&role=admin')

Ví dụ chi tiết

GET Example
fetch("https://jsonplaceholder.typicode.com/users")
  .then(response => response.json())
  .then(data => {
    console.log(data);
  })
  .catch(error => {
    console.log("Error:", error);
  });

3.2. POST - Tạo mới

Đặc điểm

  • Mục đích: Tạo resource mới trên server
  • Không idempotent: Gọi nhiều lần tạo nhiều bản ghi
  • Không cache được (thường không cache)
  • Có body chứa dữ liệu cần tạo
  • Response thường là: 201 Created + dữ liệu vừa tạo

Cú pháp

POST Request
fetch('https://api.example.com/users', {
  method: 'POST',
  body: JSON.stringify({
    name: 'Nguyễn Văn A',
    email: 'a@example.com',
    password: 'secret123'
  })
})

3.3. PUT - Cập nhật toàn bộ

Đặc điểm

  • Mục đích: Thay thế hoàn toàn một resource
  • Idempotent: Gọi nhiều lần cho kết quả giống nhau
  • Yêu cầu gửi đầy đủ fields (nếu thiếu sẽ bị xóa/mặc định)
  • Thường dùng với ID cụ thể: PUT /users/123

Ví dụ

PUT Example
// Cập nhật toàn bộ user
const updateUser = async (userId, userData) => {
  const response = await fetch(`https://api.example.com/users/${userId}`, {
    method: 'PUT',
    body: JSON.stringify({
      name: userData.name,
      email: userData.email,
      phone: userData.phone,
      address: userData.address
      // Thiếu field nào sẽ bị set về null/default
    })
  });
  
  return response.json();
};

3.4. PATCH - Cập nhật một phần

Đặc điểm

  • Mục đích: Cập nhật một hoặc vài fields của resource
  • Idempotent: Có thể idempotent tùy cách implement
  • Chỉ gửi fields cần thay đổi
  • Tiết kiệm băng thông hơn PUT

Ví dụ

PATCH Example
// Chỉ cập nhật email
const updateEmail = async (userId, newEmail) => {
  const response = await fetch(`https://api.example.com/users/${userId}`, {
    method: 'PATCH',
    body: JSON.stringify({
      email: newEmail // Chỉ gửi field cần update
    })
  });
  
  return response.json();
};

3.5. DELETE - Xóa dữ liệu

Đặc điểm

  • Mục đích: Xóa resource khỏi server
  • Idempotent: Xóa lần đầu thành công, lần sau trả về 404
  • Response thường là: 204 No Content (không có body)

Ví dụ

DELETE Example
const deleteUser = async (userId) => {
  try {
    const response = await fetch(`https://api.example.com/users/${userId}`, {
      method: 'DELETE',
      headers: {
        'Authorization': `Bearer ${token}`
      }
    });
    
    if (response.status === 204) {
      return { success: true, message: 'Xóa thành công' };
    } else if (response.status === 404) {
      return { success: false, message: 'User không tồn tại' };
    } else {
      const data = await response.json();
      return { success: false, message: data.message };
    }
  } catch (error) {
    return { success: false, message: error.message };
  }
};

3.7. OPTIONS - Kiểm tra phương thức

Đặc điểm

  • Mục đích: Hỏi server cho phép những methods nào
  • Thường dùng cho CORS (trình duyệt tự động gọi)
  • Response: Trả về header Allow: GET, POST, PUT, DELETE

Ví dụ

OPTIONS Example
// Kiểm tra API cho phép những methods nào
const checkAllowedMethods = async () => {
  const response = await fetch('https://api.example.com/users', {
    method: 'OPTIONS'
  });
  
  const allowedMethods = response.headers.get('Allow');
  console.log('Allowed methods:', allowedMethods);
  // "GET, POST, PUT, DELETE, HEAD, OPTIONS"
};

PHẦN 5: CALL API AXIOS

I. Axios là gì?

Axios là một thư viện JavaScript phổ biến, mạnh mẽ và dễ sử dụng, dùng để thực hiện các HTTP request từ:

  • Trình duyệt: Sử dụng XMLHttpRequest phía sau.
  • Môi trường Node.js: Sử dụng module http tích hợp sẵn.

Nó cung cấp một API thân thiện, dựa trên Promise, giúp việc giao tiếp với các API trở nên đơn giản và hiệu quả hơn so với việc sử dụng fetch() thông thường.

Vì sao dùng Axios thay vì Fetch?

Mặc dù fetch() là một API có sẵn trong hầu hết các trình duyệt hiện đại, Axios cung cấp nhiều tiện ích và giải pháp cho các vấn đề thường gặp, giúp code gọn gàng và dễ quản lý hơn.

Tiêu chí Fetch API Axios
Xử lý JSON ❌ Phải gọi phương thức .json() trên response. ✅ Tự động parse JSON. Dữ liệu có sẵn trong response.data.
Xử lý lỗi HTTP ❌ Chỉ reject promise khi có lỗi mạng. Lỗi HTTP (4xx, 5xx) vẫn được coi là thành công (resolve). ✅ Tự động reject promise cho các mã lỗi HTTP, dễ dàng bắt lỗi trong .catch() hoặc try/catch.
Timeout ❌ Không có cơ chế timeout tích hợp. Phải tự dùng AbortController. ✅ Hỗ trợ timeout dễ dàng qua cấu hình timeout.
Interceptor ❌ Không hỗ trợ. ✅ Có interceptor cho phép can thiệp vào request và response trước khi chúng được xử lý bởi .then() hoặc .catch() (rất hữu ích để thêm token, log lỗi toàn cục,...).
Tạo instance ❌ Không hỗ trợ. ✅ Cho phép tạo instance với cấu hình riêng (baseURL, headers mặc định,...) để tái sử dụng.
Cú pháp Tương đối dài dòng hơn. Ngắn gọn, rõ ràng hơn.

HTML với CDN

</head>
<body>
    <script>
        // Giờ bạn có thể sử dụng axios ở đây
        axios.get('https://api.example.com/data')
            .then(response => console.log(response.data));
    </script>
</body>
</html>

III. CẤU TRÚC REQUEST TRONG AXIOS

Khi bạn thực hiện một request với Axios, nó trả về một Promise. Khi Promise này được giải quyết (resolved), bạn sẽ nhận được một response object với cấu trúc đầy đủ thông tin.

Cấu trúc của Response Object:

Response Object
{
  // `data`: Phản hồi từ server đã được parse tự động (quan trọng nhất)
  data: {},
  
  // `status`: Mã trạng thái HTTP (ví dụ: 200, 201, 404, 500)
  status: 200,
  
  // `statusText`: Thông điệp trạng thái HTTP (ví dụ: "OK", "Created")
  statusText: 'OK',
  
  // `headers`: Các headers mà server gửi về
  headers: {},
  
  // `config`: Cấu hình gốc của request mà bạn đã dùng
  config: {},
  
  // `request`: Đối tượng request gốc (dùng trong môi trường Node.js hoặc XMLHttpRequest)
  request: {}
}

👉 Điều quan trọng nhất cần nhớ: Dữ liệu thực tế mà bạn cần luôn nằm trong thuộc tính response.data.

IV. CÁC METHOD HTTP TRONG AXIOS

1 GET – Lấy dữ liệu

Mục đích: Yêu cầu server gửi về một tài nguyên (danh sách, chi tiết).

Đặc điểm: Không gửi kèm dữ liệu trong body.

Ví dụ: Lấy danh sách tất cả người dùng.

Axios GET
axios.get('https://jsonplaceholder.typicode.com/users')
    .then(response => {
        // Xử lý khi thành công
        console.log('Dữ liệu nhận được:', response.data);
    })
    .catch(error => {
        // Xử lý khi thất bại (lỗi mạng, lỗi server 4xx/5xx)
        console.error('Có lỗi xảy ra:', error.message);
    });

2 POST – Tạo dữ liệu mới

Mục đích: Gửi dữ liệu mới lên server để tạo một tài nguyên.

Đặc điểm: Dữ liệu cần tạo được gửi trong body (thường là object).

Ví dụ: Thêm một người dùng mới.

Axios POST
const newUser = {
    name: 'Nguyễn Văn A',
    email: 'nguyenvana@example.com'
};

axios.post('https://jsonplaceholder.typicode.com/users', newUser)
    .then(response => {
        // Server thường trả về dữ liệu của bản ghi vừa tạo (kèm ID mới)
        console.log('Tạo thành công:', response.data);
    })
    .catch(error => {
        console.error('Lỗi khi tạo:', error.message);
    });

3 PUT – Cập nhật toàn bộ

Mục đích: Thay thế hoàn toàn một tài nguyên đã tồn tại bằng dữ liệu mới gửi lên.

Đặc điểm: Cần gửi toàn bộ các trường của tài nguyên đó.

Ví dụ: Cập nhật toàn bộ thông tin của người dùng có ID là 1.

Axios PUT
const updatedUser = {
    name: 'Trần Thị B (Đã sửa)',
    email: 'tranthib@example.com'
    // Giả sử user gốc còn có trường phone, nếu không gửi lên, nó có thể bị mất
};

axios.put('https://jsonplaceholder.typicode.com/users/1', updatedUser)
    .then(response => console.log('Cập nhật PUT thành công:', response.data))
    .catch(error => console.error('Lỗi PUT:', error));

4 PATCH – Cập nhật một phần

Mục đích: Chỉ cập nhật một vài trường cụ thể của tài nguyên.

Đặc điểm: Chỉ gửi những trường cần thay đổi.

Ví dụ: Chỉ cập nhật tên của người dùng có ID là 1.

Axios PATCH
const partialUpdate = {
    name: 'Chỉ cập nhật tên này thôi'
};

axios.patch('https://jsonplaceholder.typicode.com/users/1', partialUpdate)
    .then(response => console.log('Cập nhật PATCH thành công:', response.data))
    .catch(error => console.error('Lỗi PATCH:', error));

5 DELETE – Xóa

Mục đích: Yêu cầu server xóa một tài nguyên.

Đặc điểm: Thường không có body.

Ví dụ: Xóa người dùng có ID là 1.

Axios DELETE
axios.delete('https://jsonplaceholder.typicode.com/users/1')
    .then(response => {
        // Thành công, thường response.data có thể là rỗng hoặc thông báo
        console.log('Xóa thành công. Status:', response.status);
    })
    .catch(error => console.error('Lỗi khi xóa:', error));

V. ASYNC/AWAIT VỚI AXIOS

async/await là cú pháp hiện đại trong JavaScript giúp bạn làm việc với Promise một cách dễ dàng hơn, làm cho code trông giống như đồng bộ (synchronous) và dễ đọc, dễ hiểu hơn so với việc sử dụng .then() và .catch().

  • async được đặt trước một hàm để báo hiệu rằng hàm đó sẽ luôn trả về một Promise.
  • await được đặt trước một Promise (ví dụ: axios.get()) và nó sẽ tạm dừng việc thực thi hàm async cho đến khi Promise đó được "giải quyết" (resolved hoặc rejected). Sau đó, nó trả về kết quả của Promise đó.

Ví dụ chuẩn với try...catch (cách được khuyến nghị):

Axios với Async/Await
// Định nghĩa một hàm async
async function fetchUserData() {
    try {
        // 'await' sẽ đợi axios.get hoàn thành và gán response cho biến
        const response = await axios.get('https://jsonplaceholder.typicode.com/users/1');
        
        // Code ở đây chỉ chạy sau khi request thành công
        console.log('Dữ liệu người dùng:', response.data);
        
    } catch (error) {
        // Nếu có bất kỳ lỗi nào trong khối try (kể cả lỗi mạng, lỗi HTTP), nó sẽ được bắt ở đây
        console.error('Đã xảy ra lỗi khi gọi API:', error.message);
    }
}

// Gọi hàm
fetchUserData();

VI. GỬI QUERY PARAMS

Query parameters (thường gọi là params) là cách để thêm các thông số vào URL nhằm mục đích lọc, phân trang, tìm kiếm hoặc sắp xếp dữ liệu.

Ví dụ URL có params: https://api.example.com/users?page=2&limit=10&search=john

Với Axios, thay vì tự tay ghép chuỗi, bạn có thể sử dụng tùy chọn params trong cấu hình. Axios sẽ tự động encode và thêm chúng vào URL.

Ví dụ:

Axios với Query Params
async function getPaginatedUsers(pageNumber, pageLimit) {
    try {
        const response = await axios.get('https://jsonplaceholder.typicode.com/users', {
            params: {
                // Key: Tên param, Value: Giá trị
                page: pageNumber, // Ví dụ: 2
                limit: pageLimit, // Ví dụ: 5
                sort: 'name' // Sắp xếp theo tên
            }
        });
        
        console.log('Dữ liệu trang:', response.data);
    } catch (error) {
        console.error('Lỗi lấy dữ liệu phân trang:', error);
    }
}

getPaginatedUsers(2, 5);
// URL được gọi sẽ là: https://jsonplaceholder.typicode.com/users?page=2&limit=5&sort=name

VIII. TẠO AXIOS INSTANCE

Khi bạn gọi nhiều API đến cùng một server (ví dụ: https://api.example.com), việc lặp lại baseURL và các cấu hình chung (như timeout, headers mặc định) là không hiệu quả. Axios Instance cho phép bạn tạo ra một bản sao của Axios với các cấu hình mặc định, sau đó tái sử dụng nó.

Lợi ích:

  • Tái sử dụng cấu hình: Chỉ cần set baseURL, timeout, headers mặc định một lần.
  • Tổ chức code tốt hơn: Có thể tạo các instance khác nhau cho các phần khác nhau của ứng dụng (ví dụ: instance cho API công khai, instance cho API cần xác thực).
  • Dễ dàng thay đổi cấu hình toàn cục.
Axios Instance
const api = axios.create({
    baseURL: "https://jsonplaceholder.typicode.com",
    timeout: 5000
});

api.get("/users");
